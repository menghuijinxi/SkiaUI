# C++ 代码规范

本文档是本项目的长期 C++ 编码规范。后续补充规则时，请优先追加到对应章节；如果需要推翻旧规则，请在“规则变更记录”里写明原因。

## 1. 适用范围与优先级

本规范适用于所有 `.h`、`.hpp`、`.cc`、`.cpp`、`.cxx` 文件，以及直接影响 C++ 构建、测试和格式化的配置文件。

规则优先级从高到低：

1. 用户在当前任务中的明确要求。
2. `AGENTS.md` 中的 AI 行为要求。
3. 本文档的项目规范。
4. 当前文件或相邻模块已经稳定存在的局部风格。
5. 通用 C++ 最佳实践。

如果规则冲突，AI 必须按更高优先级执行，并在必要时说明取舍。

## 2. 默认语言与工具要求

- 默认使用 C++23；如果项目构建系统已经指定更高或更低标准，以构建系统为准。
- 新代码必须能通过项目现有编译配置。
- Windows 平台如果使用 CMake，默认使用 Visual Studio 自带的 CMake，以提高与 MSVC
  工具链和 VS 生成器的兼容性；只有在项目明确要求特定版本或 VS 自带版本不满足需求时，
  才改用其他 CMake，并说明原因。
- 不引入新的第三方库，除非用户明确同意或项目已经依赖该库。
- 优先使用标准库、RAII、值语义和清晰的数据所有权。
- 不使用编译器私有扩展，除非项目已有明确依赖。

### 编码与脚本存储

- 项目默认使用 UTF-8 编码；Windows 平台下的 C++ 编译配置也按 UTF-8 处理源码和字符串。
- 脚本文件统一保存为 UTF-8 with BOM，并使用 LF 换行，保持与编辑器状态栏显示的
  `UTF-8 with BOM`、`LF` 一致。脚本文件包括 `.ps1`、`.py`、`.bat`、`.cmd`、`.sh`、
  `.cmake` 等构建、测试、生成和辅助脚本。
- 脚本或源码中的中文字符串应直接写成人类可读的 UTF-8 文本，例如 `"中文"`；禁止为了规避
  编码问题写成 `"\xE4\xB8\xAD\xE6\x96\x87"` 这类字节转义形式，除非当前代码明确是在测试
  或构造二进制字节序列。
- 与 Windows API 交互时，应通过项目统一的转换函数在 UTF-8 字符串和 UTF-16
  `std::wstring` 之间转换；不要在业务代码中零散手写编码转换，也不要因为 Windows API
  需要宽字符而把源码或脚本改成其他编码。
- 统一转换函数表示“统一边界和入口”，不表示每次调用 Windows API 或第三方库时都必须现场
  重复转换。热路径中如果会反复使用同一个字符串，应缓存转换结果、预转换、批量转换，或让
  底层封装直接接收已经转换好的目标编码字符串。
- 高频 Windows API 模块可以在模块内部短暂维护 UTF-16 `std::wstring`，例如目录遍历、
  注册表批量访问、窗口枚举等；但 UTF-16 不应无故泄漏到通用业务层和跨平台接口。
- 第三方库适配层应尊重库自身的字符串约定：库需要 UTF-8 时保持 UTF-8，库需要 UTF-16 或
  平台原生字符串时在适配层集中转换。不要为了兼容 Windows API 把整个项目改成 UTF-16，
  否则会把转换成本转移到 JSON、网络、脚本、日志和大多数 UTF-8 第三方库边界。
- 性能敏感的字符串转换必须以实际调用频率和性能数据为依据优化；优先消除循环内重复转换，
  而不是提前牺牲全项目编码一致性。

### 第三方库与基础能力

- 图片处理、压缩/解压、网络通信、数据解析、序列化、加密校验、时间日期、路径处理等基础能力，优先寻找稳定且兼容当前需求的成熟库，不要轻易手写。
- 手写基础库级功能极易引入隐藏 bug、安全问题和兼容性问题；除非需求非常简单、范围明确，或项目明确禁止依赖，否则不应从零实现。
- 选择库时优先考虑稳定、维护活跃、经过大量生产验证、文档完整、许可证适配项目的库。
- 优先复用项目已经引入的库；如果需要新增依赖，必须先说明选择理由、替代方案、许可证和构建影响，并取得用户确认。
- 不为了“少一个依赖”而复制粘贴网上片段或临时手搓复杂协议、格式解析、压缩算法、图像编解码等能力。
- 若最终选择手写实现，必须说明为什么现有库不适用，并把实现范围控制在当前需求所需的最小集合。

## 3. 文件组织

- 头文件只放声明、轻量内联函数、模板实现和必要的常量定义。
- 源文件放具体实现，避免把复杂逻辑塞进头文件。
- 每个头文件必须有 include guard 或 `#pragma once`；若项目没有既定规则，默认使用 `#pragma once`。
- include 顺序默认如下：
  1. 当前模块对应头文件。
  2. C 标准库头文件。
  3. C++ 标准库头文件。
  4. 第三方库头文件。
  5. 项目内部头文件。
- 不使用通配式或过宽的 include；只包含当前文件真实需要的头文件。
- 头文件中禁止 `using namespace ...`。

## 4. 命名规范

若项目已有稳定命名风格，优先保持一致；没有明确风格时使用以下默认规则：

- 类型名、类名、结构体名、枚举名：`PascalCase`，例如 `UserProfile`。
- 函数名：`camelCase`，例如 `loadConfig()`。
- 局部变量和成员变量：`camelCase`。
- 私有成员变量：末尾加下划线，例如 `connectionState_`。
- 常量：`kPascalCase`，例如 `kMaxRetryCount`。
- 宏：`UPPER_SNAKE_CASE`；除 include guard 或平台适配外，尽量不用宏。
- 命名空间：小写，必要时使用下划线，例如 `game_core`。

命名必须表达业务含义，不使用 `tmp`、`data`、`obj`、`manager` 这类含糊名称，除非作用域极小且含义明显。

## 5. 格式化与排版

- 缩进使用 4 个空格，不使用 tab。
- 单行尽量不超过 100 字符；超过时按语义换行。
- 大括号风格：函数、类、控制语句的大括号独占或同行，以项目现有格式为准；没有现有格式时使用 K&R 风格。
- `if`、`for`、`while` 等控制语句即使只有一行，也必须写大括号。
- 指针和引用符号靠近类型或变量时，跟随当前文件风格；新文件默认写作 `Type* ptr`、`Type& ref`。
- 相关代码块之间留一个空行；不要用多个空行制造视觉分隔。

## 6. 类型与接口设计

- 接口应表达最小必要能力，不暴露无关实现细节。
- 优先使用强类型表达业务概念，避免用裸 `int`、`string` 传递含义复杂的数据。
- 函数参数超过 4 个时，优先考虑参数对象或拆分职责。
- 只读入参优先使用 `const T&`；小型平凡类型按值传递。
- 需要转移所有权时使用 `std::unique_ptr`；共享所有权必须有明确理由才使用 `std::shared_ptr`。
- 不返回悬垂引用、悬垂指针或指向局部对象的地址。
- 对外接口尽量稳定，修改公共接口时同步更新调用方和测试。
- 需要暴露给外部访问的成员变量，禁止直接作为公共变量暴露；变量本身统一放到
  `protected` 区域，并通过 `SetXxx()` 和 `GetXxx()` 这类访问函数对外提供读写能力，
  便于未来增加校验、日志、断点调试或兼容逻辑。
- 仅做简单赋值或返回的 `SetXxx()`、`GetXxx()` 函数应直接在 `.h` 的类定义内实现，
  让调用方有机会内联；不要为了这类短小访问函数批量手动标记 `inline`，现代编译器
  会根据优化策略自动决定是否内联。
- 当 `SetXxx()`、`GetXxx()` 已经包含非平凡校验、状态联动、资源操作或复杂逻辑时，
  应将实现放入源文件，并优先保证接口行为清晰和二进制边界稳定。
- 对外暴露的函数可以对指针、句柄、可选对象等输入做谨慎的空值检查，并给出明确错误处理。
- 内部函数默认不要频繁判断空。内部重复空判断容易掩盖调用方错误，使 bug 被静默忽略。
- 内部函数的非空前置条件优先通过引用、非空封装类型、清晰命名或调用约定表达；必要时可以使用断言暴露错误。
- 除非空值是该内部流程的合法业务状态，否则内部代码大多数情况下不应额外判断空。

### 封装与复用

- 实现功能前先查找是否已有可复用函数；优先调用已有实现，不要重复写相同逻辑。
- 重复逻辑需要封装成函数，即使只是两三句代码，也应优先抽出清晰的小函数。
- 不要因为担心函数数量影响性能而拒绝封装；现代 C++ 编译器通常能自动内联小函数。性能优化必须基于实际证据。
- 封装函数的目的必须清晰单一，名称要准确表达它完成的动作或业务含义。
- 不要把多套不同逻辑混在同一个函数里，也不要用大量布尔参数、模式参数或分支把不相关流程揉到一起。
- 如果封装后函数参数过多，说明抽象边界可能不对；应考虑拆成多个职责更小的函数，或使用有明确语义的参数对象。
- 内部辅助函数应放在合适的最小作用域内，例如源文件匿名命名空间、类的私有成员函数或局部 helper，不要无故扩大为公共接口。

### 模块化与职责划分

- 不要把所有代码逻辑都写进一个脚本、一个源文件、一个类或一个超长函数里。
- 应根据功能和职责拆分为多个类、模块或 helper，让每个单元各司其职。
- 每个类或模块都应该有清晰边界：它负责什么、不负责什么，应能从命名和接口看出来。
- 功能需求复杂时，应主动抽象出合适的概念，例如策略、状态、处理器、命令、调度表或领域对象。
- 不要用几十个、上百个 `switch` 和 `if` 分支堆叠复杂业务逻辑。若分支代表不同类型、状态、行为或业务规则，应优先考虑多态、策略表、状态机、查表驱动或拆分处理器。
- 简单局部分支可以保留；抽象必须服务于降低复杂度，不要为了形式制造空洞类。
- 拆分后的类和函数必须协作清晰，不能把原本混乱的逻辑换成分散但更难追踪的间接调用。

## 7. 资源管理与内存安全

- 禁止手写裸 `new` / `delete`，除非是在实现底层封装，并且没有标准库替代。
- 使用 RAII 管理文件、锁、网络连接、句柄和内存。
- 容器优先使用 `std::vector`、`std::array`、`std::unordered_map`、`std::map` 等标准容器。
- 不使用 C 风格数组作为可变长度容器。
- 不使用 `memset`、`memcpy` 操作非平凡 C++ 对象。
- 所有权必须能从类型上看出来；不能只靠注释解释谁负责释放资源。

## 8. 错误处理

- 错误处理策略必须和当前模块一致。
- 不吞掉错误。捕获异常后必须处理、转换或补充上下文再向上传递。
- 不用返回魔法值表示复杂错误；优先使用明确的结果类型、错误码枚举或异常。
- 失败路径必须保持对象处于可析构、可恢复或明确失效的状态。
- 日志不能替代错误处理。

## 9. const、constexpr 与状态

- 能标记为 `const` 的变量、参数和成员函数应标记为 `const`。
- 编译期常量优先使用 `constexpr`。
- 避免全局可变状态。必须使用时，要说明生命周期、线程安全和初始化顺序。
- 类的不变量必须由构造函数建立，并由成员函数维护。

## 10. 并发与线程安全

- 共享可变状态必须有明确同步策略。
- 使用锁时优先使用 `std::lock_guard`、`std::unique_lock` 等 RAII 锁类型。
- 不在持锁期间执行未知耗时操作、回调或外部 I/O，除非设计上明确要求。
- 原子变量只用于简单同步；复杂状态不要仅靠多个原子变量拼接。
- 新增异步逻辑必须说明任务生命周期和取消方式。

## 11. 注释与文档

- 注释应该谨慎、克制，但不能完全不写。
- 注释解释“为什么”，不要重复代码已经表达的“做什么”。
- 复杂算法、协议边界、线程模型、所有权转换必须有简短说明。
- 晦涩难懂、容易被误解、后续维护可能产生偏差的代码，应写注释说明关键意图和约束。
- 不要写教程式注释，不要把每一句普通代码都解释一遍。
- 不要对第三方库的每句普通调用都写注释；只有调用顺序、参数约束、生命周期、线程要求或库行为不直观时才解释。
- 注释语言应与使用者和 AI 交流的语言一致；本项目当前默认使用中文注释，避免使用者无法阅读注释而难以参与修改。
- TODO 必须包含原因和后续动作，例如 `TODO: replace with streaming parser after protocol v2 lands`。
- 未来才实现的功能函数、占位函数、临时空实现或故意保留的能力缺口，必须标记 TODO。
- 这类 TODO 必须说明以后会怎么实现，以及为什么现在不实现，不能只写 `TODO` 或 `implement later`。
- 不保留无效注释、注释掉的旧代码或调试输出。

## 12. 测试要求

- 修复 bug 时应添加能失败再通过的回归测试，除非项目没有测试框架或代价明显不合理。
- 新增公共行为时必须覆盖正常路径、边界条件和失败路径。
- 测试命名应表达行为，不只描述实现细节。
- 不为了让测试通过而降低断言质量。
- 测试用临时文件、调试导出、生成样例和一次性验证产物统一放到项目根目录 `tmp/` 下。
- 不要在项目根目录或业务目录里散落创建测试文件、测试文件夹和导出文件，避免污染 git 状态。
- `tmp/` 只用于本地临时产物，不提交到 git；需要长期保留的测试 fixture 应放到正式测试目录，并使用清晰命名。
- 任务结束前应清理不再需要的临时产物；如果保留在 `tmp/` 是为了用户检查，最终回复里说明。
- 如果无法运行测试，最终回复必须说明未验证项。

## 13. AI 写代码流程

AI 在处理 C++ 任务时必须遵循：

1. 先阅读相关文件和本规范，不凭空假设架构。
2. 明确当前改动属于 bug 修复、功能新增、重构还是格式调整。
3. 除非用户明确要求修改代码规范或 AI 行为规则，否则禁止修改 `AGENTS.md` 和
   `docs/cpp-code-standard.md`。如果 AI 认为需要补充规范，应在最终回复中提出建议，
   或按问题复盘规则写入独立分类文档，不得自行改动规范文件。
4. 优先做最小可行改动，保持行为边界清晰。
5. 遇到重大技术分支时必须暂停实现，与用户确认方向后再继续。重大技术分支包括：
   大规模新增或改写代码、架构或模块边界变化、公共接口设计变化、数据格式或协议变化、
   持久化方案变化、第三方库选型、线程模型变化、性能与兼容性取舍明显的方案，以及任何
   一旦走错会导致大量返工或长期维护成本的决策。
6. 暂停确认时应清楚说明当前目标、可选方案、各方案影响、推荐方向和主要风险；在用户确认
   前，只做阅读、定位、原型验证或不改变长期方向的最小准备工作。
7. 如果用户已经明确授权 AI 自主决定技术方案，仍需在最终回复中说明关键取舍；但遇到明显
   超出原任务边界或风险显著增加的分支时，仍应重新停下来确认。
8. 新增或修改接口时同步更新调用方、测试和必要文档。
9. 功能设计发生变动后，默认不推断用户需要兼容以前版本或旧逻辑；只有用户明确要求兼容、
   已有公开接口契约要求兼容，或迁移方案明确要求保留时，才保留兼容代码。否则应删除已经用不上的
   旧实现、旧分支、旧配置和临时适配层，保持最终代码精简；不要为了“可能以后用到”默认保留死代码。
   删除后，最终回复要明确提醒用户删掉了哪些分支逻辑或兼容路径。
10. 完成后运行可用的格式化、构建或测试命令。
11. 最终回复说明改了什么、验证了什么、还有什么未验证。

### 崩溃与异常行为诊断流程

出现崩溃、异常返回、行为不符合预期或性能明显异常时，AI 必须先诊断再修改，禁止盲目改代码。

1. 优先查看日志、错误信息、崩溃栈、断言位置、返回码和最近的运行上下文。
2. 先检查是否由上一次修改造成。阅读最近 diff，确认改动范围和可能影响的调用路径。
3. 通过临时注释、临时禁用或最小化上一次修改的代码来缩小范围，定位到具体函数、分支或语句。
4. 临时注释只用于诊断，不能把大段注释代码作为最终结果留下。
5. 找到具体出错代码后，如果是简单错误，例如空指针、越界、条件写反、生命周期错误、参数错误，应直接修复并补充验证。
6. 如果问题涉及第三方库，优先阅读官方文档、示例和限制说明，再阅读源码或 issue 记录。
7. 使用第三方库出错时，重点核对参数是否合法、调用顺序是否正确、对象生命周期是否满足要求、线程约束是否被违反、错误码是否被正确处理。
8. 如果当前改动无法解释问题，需要阅读 git 提交记录、`git blame` 和相关历史 diff，判断错误是否更早引入。
9. 如果历史上修复过类似问题，需要找到当时的修复方式，判断是旧修复不彻底，还是后来改动破坏了旧修复。
10. 错误修复后，必须回看为诊断和验证添加的大量代码，区分哪些是真正修复所需，哪些只是临时测试、日志、导出、探针或调试分支。
11. 临时测试代码、调试日志、一次性验证路径和无用分支必须清理，不要把诊断过程中的冗余代码留在最终实现里。
12. 清理后必须重新运行相关测试或最小复现验证，确认结果与清理前一致，才能认为问题修复完成。
13. 修复完成后，应补充能覆盖本次问题的回归测试或最小复现验证，避免同类错误再次出现。

### 问题复盘与知识沉淀

AI 在解决有长期参考价值的问题后，应主动把经验沉淀到文档中，便于以后通过阅读文档快速定位同类问题。

- 需要沉淀的情况包括：根因隐蔽、诊断过程复杂、涉及重大技术取舍、容易复发、影响范围较大、
  第三方库用法不直观、踩到平台或编译器差异、性能问题定位成本高，或解决方案对后续架构有指导价值。
- 不要把每个普通小改动都写入复盘文档；只有对后续排查、设计或选型有复用价值时才记录。
- 复盘文档统一放到 `docs/knowledge/` 下，并按问题类型拆分为多个文档，例如
  `crash-and-exception.md`、`performance.md`、`architecture.md`、`third-party-libraries.md`、
  `build-and-toolchain.md`、`testing-and-debugging.md`。如果已有文档能容纳当前经验，优先追加到
  现有文档；否则再新增分类文档。
- 每条复盘应包含：现象、影响范围、根因、排查路径、最终解决方案、验证方式、可复用经验、
  后续遇到类似问题时的快速检查清单。
- 第三方库相关复盘应额外记录：库版本、关键 API 用法、参数和生命周期约束、线程要求、错误码或异常处理方式、
  文档链接或本地参考位置，以及本次踩到的坑和规避方式。
- 如果复盘涉及尚未完全确认的推断，必须明确标注“推断”或“待验证”，不要把猜测写成确定结论。
- 完成有价值的问题修复后，最终回复中应说明是否已更新知识沉淀文档；如果没有更新，应简要说明原因。
- AI 自己的经验补充、问题总结和第三方库踩坑记录，应按照本节要求形成独立分类文档；
  除非用户明确要求修改规范，不得把这类内容直接追加到 `AGENTS.md` 或 `docs/cpp-code-standard.md`。

### Windows 拉取项目与代理

- Windows 平台拉取项目或更新依赖时，如果 `git clone`、`git fetch`、`git pull`、子模块更新等操作
  出现网络连接超时，应先检查 Windows 系统代理是否开启。
- 如果系统代理已开启，应读取系统代理配置中的服务器地址和端口，并优先通过当前命令会话的
  `HTTP_PROXY` / `HTTPS_PROXY` 环境变量，或 Git 的临时 `http.proxy` / `https.proxy` 配置使用该代理
  重新拉取。
- 代理配置通常属于本机环境信息，不要把具体代理地址、端口、账号或密码写入仓库脚本、正式文档或提交内容；
  除非用户明确要求持久化，否则只对当前拉取命令临时生效。
- 如果系统代理未开启，或使用系统代理后仍然超时，应在回复中说明已检查的代理状态、使用的拉取方式和剩余错误信息。

### 本地环境配置记录

项目环境配置、工具链路径、IDE 设置、SDK 安装位置、缓存目录、环境变量、端口占用、代理配置等通常与单台设备强相关，不适合提交到 git。

- 多个 AI 会话之间需要交接本地环境信息时，应统一记录到项目根目录 `tmp/local-environment/`
  下的 Markdown 文档中，例如 `tmp/local-environment/windows-msvc.md` 或
  `tmp/local-environment/vcpkg.md`。
- 这类文档只记录本机排查和复现所需的信息，不写入账号、令牌、私钥、密码、内部地址等敏感内容。
- 如果环境问题具有跨设备通用价值，例如 CMake 参数含义、依赖安装原则或第三方库通用限制，
  应拆分为可提交的正式文档；设备相关路径和临时状态仍保留在 `tmp/local-environment/`。
- 任务结束时，如果创建或更新了本地环境配置记录，最终回复中应说明记录文件路径和用途。
